# AGENTS.md

Barbershop booking app built with Next.js 15 (App Router), Prisma + PostgreSQL, next-auth (Google), and Tailwind/shadcn-style UI components. No tests, no CI.

## Commands

```bash
npm install          # `prepare` postinstall runs `prisma generate`
npm run dev          # Next dev server on :3000
npm run build
npm run lint         # `next lint` — ESLint with Rocketseat config + simple-import-sort
```

There is **no typecheck script**. TypeScript is checked implicitly via `next build` (`tsconfig` is `noEmit`, strict). To typecheck without a full build: `npx tsc --noEmit`.

Prisma / DB:

```bash
docker compose up -d                      # start Postgres (container fsw-v3-postgres, user/pass: postgres/password, :5432)
npx prisma migrate dev                    # apply/create migrations in dev
npx prisma db seed                        # seed 10 barbershops + services (seed is CommonJS via ts-node)
npx prisma studio
```

## Required env vars (create `.env` — gitignored)

- `DATABASE_URL` — e.g. `postgresql://postgres:password@localhost:5432/barbershop?schema=public`
- `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` — Google OAuth credentials
- `NEXT_AUTH_SECRET` — next-auth session secret

Without these, auth + DB fail at runtime.

## Architecture notes that aren't obvious from filenames

- Route group `app/(home)/` holds the root marketing/search pages without affecting the URL.
- Prisma client is instantiated once in `app/_lib/prisma.ts` and cached on `globalThis` in dev to avoid connection exhaustion under HMR. Import the client as `import { db } from '@/app/_lib/prisma'` — never construct `new PrismaClient()` elsewhere.
- Server actions live in `app/_actions/` (e.g. `cancel-booking.ts`). Import them into server components / `use server` boundaries; do not call them from client components directly.
- next-auth config is in `app/_lib/auth.ts`; the route handler is `app/api/auth/[...nextauth]/route.ts`. The session callback injects `user.id` onto `session.user` — rely on this rather than re-querying.
- Prisma schema models: `User`, `Barbershop`, `Service`, `Booking`, plus next-auth's `Account`/`Session`/`VerificationToken`.
- Images for barbershops/services are hosted on `utfs.io` (allowlisted in `next.config.mjs`). Seed data references specific `utfs.io` URLs.
- App is **dark-only**: `<body className="dark ...">` is hardcoded in `app/layout.tsx`. `next-themes` is installed but the toggle wiring is manual — don't assume light mode works without checking.

## Lint/style conventions (enforced as errors)

- `eslint-plugin-simple-import-sort` is on; import order **must** be sorted or `npm run lint` fails. Keep imports grouped: builtin → external → internal (`@/*`).
- Path alias `@/*` → repo root (configured in both `tsconfig.json` and likely `components.json`). Prefer `@/app/...` over relative imports for deep paths.
- ESLint extends `@rocketseat/eslint-config/next` + `next/core-web-vitals`; `no-var` and `@typescript-eslint/no-var-requires` are turned **off** (the seed file uses CommonJS `require`).
- Do **not** add comments unless explicitly requested.

## Gotchas

- `package-lock.json` is the lockfile — use `npm`, not yarn/pnpm, to avoid dual-lockfile drift.
- `.next/` and `.postgres-data/` are gitignored build/data dirs; never commit them.
- `next-env.d.ts` is gitignored and regenerated by Next — leave it alone.
- After pulling schema changes, run `npx prisma generate` (or `npm install`, which triggers `prepare`) before typechecking.
